# Schedule Equity Model — Design Spec

> Status: **approved** (2026-04-22)
> Author: Brainstorm session with Claude (Opus 4.7)
> Previous related work: UX improvements to `AutoGenerateDialog` (plan `2026-04-21-schedule-generator-ux-improvements.md`, merged).

## Goal

Transform App Horarios from "an auto-scheduler that counts total hours" into "an auto-scheduler that balances workload across the dimensions employees actually care about" — specifically Sundays, Saturdays, night shifts, holidays, total hours, and continuous rest blocks. Additionally, model **overtime as a first-class concept** so the algorithm fills operational demand even when doing so requires exceeding contractual caps, provided an admin approves the exception.

## Success Criteria

1. A manager running auto-generate on month *N+1* sees a noticeable bias toward employees who received fewer Sundays/holidays in months *N-2..N*.
2. No slot goes unfilled unless it is literally impossible (every candidate violates an inviolable legal/safety rule, or all are in time-off).
3. When a slot is filled using overtime, it is clearly flagged both in the auto-gen dialog preview and in the schedule grid, with a dedicated approval workflow.
4. An admin can define contract types ("Full-time", "Part-time", "Fin de semana") with per-type hard caps on Sundays and holidays per quarter, and assign employees to them.
5. The trigger-maintained `employee_equity_rollups` table always reflects the current state of `schedule_entries` (verified by SQL tests).

## Scope and Priorities

### Client profile

- **Target customer**: multi-sede single-tenant (3-10 sedes, 50-200 empleados). Moving off Excel today.
- **Product quality bar**: "producto completo" — shippable to a real client, usable without the developer at hand.
- **Top priority**: cover operational demand (**A** in user ranking).
- **Second priority**: equity between employees (**B**).

### Equity dimensions (user-ranked order)

1. Sundays worked
2. Saturdays worked
3. Night shifts worked (definition per §3.2)
4. Holidays worked
5. Total hours
6. Consecutive rest-day blocks (block-packing preference)
7. "Bad" specific shifts — *deferred to future spec*
8. Opener/closer — *deferred to future spec*

### Personas

- **Admin central (role='admin')**: edits contract types, holidays, scoring weights, and approves overtime across all sedes.
- **Manager de sede (role='manager')**: generates monthly schedules for their own sede, approves overtime for their own sede, resolves swap requests.
- **Empleado (role='employee')**: views their own published schedule and their own equity rollups.

## 1 — Architecture

Four new pieces + three existing pieces modified:

**New:**
1. **Reference data tables**: `holidays` (Colombian national pre-loaded, editable per-sede) and `contract_types` with hard caps and soft targets.
2. **Materialized rollups**: table `employee_equity_rollups` with one row per `(employee_id, year, month)`, maintained by Postgres trigger on `schedule_entries`.
3. **Shift metadata**: column `is_night BOOLEAN` on `shift_templates`, default suggested by the CST rule (shift overlaps 21:00–06:00) at template-creation time, admin-editable.
4. **Employee contract assignment**: column `contract_type_id UUID NOT NULL` on `profiles` with a seeded "Sin definir" default.

**Existing modified:**
1. `src/lib/schedule-generator.ts` — extended scoring, two-pass assignment (with/without overtime), hard-cap enforcement, new warning kinds, block-packing scoring.
2. `src/components/schedule/auto-generate-dialog.tsx` — extended preview with equity columns, overtime summary card, inline approval hand-off.
3. `src/app/(authenticated)/employees/page.tsx` — contract column + equity rollup side panel.

### Data flow

```
schedule_entries (INSERT/UPDATE/DELETE)
    ↓ trigger
employee_equity_rollups (upsert: recompute month)
    ↓ read during
schedule-generator.ts (scoring + hard caps + overtime)
    ↓ produces
AutoGenerateDialog (preview with new metrics + overtime tally)
    ↓ on apply
schedule_entries → loop closes
```

### Separation of concerns

- `holidays`, `contract_types`, `scoring_weights` → **configuration data**: edited by admin, changes infrequently.
- `employee_equity_rollups` → **derived history**: never edited by hand; always a pure function of `schedule_entries + holidays + shift_templates`.
- `shift_templates.is_night`, `profiles.contract_type_id` → **profile metadata**: edited occasionally.
- `schedule_entries.exceeds_caps`, `overtime_status`, `overtime_reviewed_*` → **per-entry state**: mutated by generator and by approval workflow.

### 2.8 Trigger — `recompute_equity_rollup`

Function that **re-aggregates** a full `(employee_id, year, month)` cell from scratch (idempotent, not delta-based). Triggered `AFTER INSERT/UPDATE/DELETE` on `schedule_entries`. On `UPDATE`, fires for both the OLD and NEW `(employee_id, date)` if either changed.

```sql
CREATE OR REPLACE FUNCTION recompute_equity_rollup(
  p_employee_id UUID,
  p_year INT,
  p_month INT
) RETURNS VOID LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  INSERT INTO employee_equity_rollups (
    employee_id, year, month,
    sundays_worked, saturdays_worked, nights_worked, holidays_worked, total_hours
  )
  SELECT
    p_employee_id, p_year, p_month,
    COUNT(*) FILTER (WHERE EXTRACT(DOW FROM se.date) = 0),
    COUNT(*) FILTER (WHERE EXTRACT(DOW FROM se.date) = 6),
    COUNT(*) FILTER (WHERE st.is_night = true),
    COUNT(*) FILTER (WHERE EXISTS (
      SELECT 1 FROM holidays h
      WHERE h.date = se.date
        AND (h.location_id IS NULL OR h.location_id = (
          SELECT s.location_id FROM schedules s WHERE s.id = se.schedule_id
        ))
    )),
    COALESCE(SUM(
      EXTRACT(EPOCH FROM (se.end_time - se.start_time +
        CASE WHEN se.end_time < se.start_time THEN INTERVAL '1 day' ELSE INTERVAL '0' END
      )) / 3600
    ), 0)
  FROM schedule_entries se
  LEFT JOIN shift_templates st ON st.id = se.shift_template_id
  WHERE se.employee_id = p_employee_id
    AND EXTRACT(YEAR FROM se.date) = p_year
    AND EXTRACT(MONTH FROM se.date) = p_month
  ON CONFLICT (employee_id, year, month) DO UPDATE SET
    sundays_worked   = EXCLUDED.sundays_worked,
    saturdays_worked = EXCLUDED.saturdays_worked,
    nights_worked    = EXCLUDED.nights_worked,
    holidays_worked  = EXCLUDED.holidays_worked,
    total_hours      = EXCLUDED.total_hours,
    updated_at       = now();
END;
$$;

CREATE OR REPLACE FUNCTION trg_recompute_rollup()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  IF TG_OP = 'DELETE' THEN
    PERFORM recompute_equity_rollup(OLD.employee_id,
      EXTRACT(YEAR FROM OLD.date)::INT,
      EXTRACT(MONTH FROM OLD.date)::INT);
    RETURN OLD;
  END IF;

  -- INSERT or UPDATE
  PERFORM recompute_equity_rollup(NEW.employee_id,
    EXTRACT(YEAR FROM NEW.date)::INT,
    EXTRACT(MONTH FROM NEW.date)::INT);

  -- UPDATE that changed employee or date: also refresh OLD bucket
  IF TG_OP = 'UPDATE' AND (
    OLD.employee_id <> NEW.employee_id OR OLD.date <> NEW.date
  ) THEN
    PERFORM recompute_equity_rollup(OLD.employee_id,
      EXTRACT(YEAR FROM OLD.date)::INT,
      EXTRACT(MONTH FROM OLD.date)::INT);
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER schedule_entries_rollup_trigger
  AFTER INSERT OR UPDATE OR DELETE ON schedule_entries
  FOR EACH ROW EXECUTE FUNCTION trg_recompute_rollup();
```

### 2.9 Trigger — holidays change cascades

When `holidays` changes, all rollups for entries on the affected dates need recomputing.

```sql
CREATE OR REPLACE FUNCTION trg_holidays_cascade()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  rec RECORD;
  affected_date DATE;
BEGIN
  affected_date := COALESCE(NEW.date, OLD.date);
  FOR rec IN
    SELECT DISTINCT
      se.employee_id,
      EXTRACT(YEAR FROM se.date)::INT  AS yr,
      EXTRACT(MONTH FROM se.date)::INT AS mo
    FROM schedule_entries se
    WHERE se.date = affected_date
  LOOP
    PERFORM recompute_equity_rollup(rec.employee_id, rec.yr, rec.mo);
  END LOOP;
  RETURN COALESCE(NEW, OLD);
END;
$$;

CREATE TRIGGER holidays_cascade_trigger
  AFTER INSERT OR UPDATE OR DELETE ON holidays
  FOR EACH ROW EXECUTE FUNCTION trg_holidays_cascade();
```

```sql
ALTER TABLE contract_types ENABLE ROW LEVEL SECURITY;

-- employee_equity_rollups: read only (trigger writes via SECURITY DEFINER)
ALTER TABLE employee_equity_rollups ENABLE ROW LEVEL SECURITY;
CREATE POLICY rollups_read ON employee_equity_rollups FOR SELECT USING (
  get_user_role() = 'admin'
  OR (get_user_role() = 'manager'
      AND employee_id IN (SELECT id FROM profiles WHERE location_id = get_user_location_id()))
  OR employee_id = auth.uid()
);
-- no INSERT/UPDATE/DELETE policy: only the SECURITY DEFINER function can write.
```

## 3 — Algorithm Changes

### 3.1 New signature

```ts
// src/lib/schedule-generator.ts

  existingEntries: ScheduleEntry[],

  // NEW

): AutoGenResult
```

The dialog fetches `rollups`, `holidays`, `contractTypes`, `scoringWeights` before invoking the generator. The generator remains pure (no Supabase calls).

### 3.2 Two time windows

| Purpose | Window | Source |
|---|---|---|
| Soft scoring (who is "behind"?) | Rolling **last 3 months** including current | Sum of rollups for `[target_month - 2, target_month - 1, target_month]` |
| Hard cap enforcement | **Calendar quarter** of the slot's date | Sum of rollups for the 3 months of that quarter |

Helpers implemented in `src/lib/equity-helpers.ts`:

```ts
export function getQuarter(year: number, month: number): number;
// e.g. getQuarter(2026, 4) → 2

export function getQuarterRange(date: string): { year: number; months: number[] };
// e.g. getQuarterRange('2026-04-15') → { year: 2026, months: [4, 5, 6] }

export function getRollingWindow(year: number, month: number, size: number):
  Array<{ year: number; month: number }>;
// e.g. getRollingWindow(2026, 4, 3) → [{2026,2},{2026,3},{2026,4}]

export function sumRollupField(
  rollups: EmployeeEquityRollup[],
  employeeId: string,
  window: Array<{ year: number; month: number }>,
  field: keyof EmployeeEquityRollup
): number;

export function isHoliday(date: string, locationId: string, holidays: HolidayDate[]): boolean;

export function isNightShift(template: ShiftTemplate): boolean;
// reads template.is_night directly (no recomputation)

export function suggestIsNight(startTime: string, endTime: string): boolean;
// heuristic: does the time range overlap 21:00-06:00?
```

### 3.3 Two-pass assignment

```ts
for (const slot of demandSlots) {
  const candidates = filterCandidatesInviolable(slot, ...);

  // Pass 1: strict — no overtime, no cap violations
  let chosen = scoreAndPick(candidates, slot, { allowOvertime: false, ... });

  // Pass 2: relaxed — allow overtime, still respects inviolables
  if (!chosen) {
    chosen = scoreAndPick(candidates, slot, { allowOvertime: true, ... });
    if (chosen) {
      chosen.exceedsCaps = computeExceededCaps(chosen, slot, rollups, constraints, contractTypes);
      chosen.overtimeStatus = 'pending';
    }
  }

  if (!chosen) {
    warnings.push({ kind: 'no_safe_candidate', positionId: slot.positionId,
                    date: slot.date, shiftTemplateId: slot.shiftTemplateId });
    continue;
  }

  entries.push(chosen);
  updateTracker(tracker, chosen);
  updateRollupLocal(localRollups, chosen);  // for subsequent slots in same run
}
```

### 3.4 Inviolable checks (Pass 1 AND Pass 2)

These never yield, even with admin override during auto-gen:

| Check | Implementation |
|---|---|
| `daily_rest` | `hasEnoughRest(lastEndTime, lastDate, slot.startTime, slot.date, requiredRestHours(lastWasNight))` |
| `daily_max_hours` | `if (slot.durationHours > constraints.maxHoursPerDay) continue` |
| `double_shift` | `if (tracker.assignedDates.has(slot.date)) continue` |
| `rest_24h_after_night` | embedded in `requiredRestHours(lastWasNight)` → 24 if last was night, 12 otherwise |
| `time_off` | `if (timeOffMap.get(empId)?.has(slot.date)) continue` |

```ts
function requiredRestHours(lastShiftWasNight: boolean, constraints: LaborConstraints): number {
  return lastShiftWasNight ? 24 : constraints.minRestHoursBetweenShifts;
}
```

### 3.5 Contractual caps (Pass 1 only)

Pass 1 additionally excludes candidates who would violate **any** of:

- `weekly_hours`: `tracker.weeklyHours[week] + slot.durationHours > min(constraints.maxHoursPerWeek, contract.target_hours_per_week ?? global, employee.max_hours_per_week)`
- `consecutive_days`: `tracker.consecutiveDays + 1 > constraints.maxConsecutiveDays`
- `sundays_quarter` (only if `dayOfWeek(slot.date) === 0`): `sumRollupField(...quarter, 'sundays_worked') + localIncrement + 1 > contract.max_sundays_per_quarter`
- `holidays_quarter` (only if holiday): similar with `holidays_worked`
- `night_limit` (only if shift is night): `sumRollupField(...month, 'nights_worked') + localIncrement + 1 > contract.target_nights_per_month` (only if `target_nights_per_month` is not null)

### 3.6 Extended scoring

Applied in Pass 1 and Pass 2 identically. In Pass 2, the cap violations contribute additional penalty so the algorithm still prefers least-bad candidate.

```ts
function scoreCandidate(
  employee, slot, tracker, rollingRollup, contract, weights
): number {
  let score = 0;

  // Position fit
  score += employee.position_id === slot.positionId
    ? weights.position_primary_bonus
    : weights.position_secondary_bonus;

  // Hour/shift deficit (existing)
  score += (targetHours - tracker.totalHours) * weights.hour_deficit_multiplier;
  score += (targetShifts - tracker.totalShifts) * weights.shift_deficit_multiplier;

  // Equity penalties (new)
  const dow = dayOfWeek(slot.date);
  if (dow === 0) score -= rollingRollup.sundays   * weights.sunday_penalty;
  if (dow === 6) score -= rollingRollup.saturdays * weights.saturday_penalty;
  if (isNightShift(slot.template)) score -= rollingRollup.nights * weights.night_penalty;
  if (isHoliday(slot.date, locationId, holidays))
    score -= rollingRollup.holidays * weights.holiday_penalty;

  // Block-packing (new)
  const gap = tracker.lastShiftDate
    ? daysBetween(tracker.lastShiftDate, slot.date)
    : null;
  const BLOCK_LENGTH_CAP = 4;
  if (gap === 1 && tracker.consecutiveDays < BLOCK_LENGTH_CAP) {
    score += weights.block_continuation_bonus;
  } else if (gap === 2) {
    score -= weights.fragmentation_penalty;
  } else if (gap !== null && gap >= 3) {
    score += weights.clean_restart_bonus;
  }

  return score;
}
```

### 3.7 New warning kinds

Extend the `AutoGenWarning` discriminated union in `src/lib/types.ts`:

```ts
export type AutoGenWarning =
  | { kind: "no_employees_in_position";  positionId: string; date: string; shiftTemplateId: string }
  | { kind: "no_available_employee";      positionId: string; date: string; shiftTemplateId: string }
  | { kind: "no_safe_candidate";          positionId: string; date: string; shiftTemplateId: string }
  | { kind: "overtime_assigned";          positionId: string; date: string; shiftTemplateId: string;
                                          employeeId: string; caps: string[] }
  | { kind: "no_templates_selected" }
  | { kind: "no_employees_selected" };
```

`no_safe_candidate` replaces the old `no_available_employee` for the case where **even Pass 2** can't find a candidate (inviolables block everyone). `overtime_assigned` is new — emitted for every slot filled in Pass 2, for preview tallying.

### 3.8 Degradation when no rollups exist

On first deployment or first run, `employee_equity_rollups` is empty → all `sumRollupField` calls return 0 → scoring has no equity signal → assignment is fair by position-fit + block-packing only. This is correct and expected.

## 5 — Error Handling and Edge Cases

### 5.1 Unfillable slot (`no_safe_candidate`)

Emitted when even Pass 2 (overtime-allowed) cannot find a candidate — every employee would violate an inviolable rule or is on time-off. Warning card in the dialog is red with copy: *"N ranuras imposibles de cubrir: todos los empleados violarían una restricción legal (descanso mínimo, doble turno, 24h tras noche, o tiempo libre aprobado)."*

### 5.2 Employee without real contract (`Sin definir`)

Algorithm runs normally for them, soft equity scoring applies, but hard caps cannot bind (caps = 999). UI surfaces the red badge in `/empleados`. Auto-gen dialog shows a pre-flight banner: *"N empleados con tipo 'Sin definir'. Asigna un tipo para aplicar restricciones de equidad. [Ir a empleados]"*.

### 5.3 Holidays changed post-generation

Trigger (§2.9) recomputes affected rollups automatically. If this pushes someone's quarter rollup past their cap, a banner in `/empleados` and `/schedule` reads: *"N empleados exceden su cap trimestral por un cambio reciente en festivos. [Revisar]"*. No auto-modification of existing entries — only surface the conflict.

### 5.4 First deployment (no historical rollups)

`employee_equity_rollups` empty → scoring falls back to position-fit + block-packing + hour-target only → fair for everyone on day 1 because no one has history to lean against. Banner on first auto-gen run: *"Primera corrida con equidad activa. El algoritmo aprenderá a balancear mejor en los próximos meses."*

### 5.5 Backfill on migration

Migration `021_backfill_equity_rollups.sql` computes rollups for all existing entries so the first post-upgrade auto-gen already has historical context. Implementation: loop over distinct `(employee_id, year, month)` in `schedule_entries` and call `recompute_equity_rollup` for each.

### 5.6 RLS leakage testing

After each policy change (§2.10), run SQL assertions (in `supabase/tests/rls_equity.sql`) verifying:
- Employee sees only their own rollups.
- Manager sees only their location's employees' rollups.
- Admin sees all.

### 5.7 Overtime approval by role

- Admin: approves any entry, any sede.
- Manager: approves entries whose `schedule_id` belongs to a schedule whose `location_id = get_user_location_id()`.
- Employee: no approval capability (UI hides the buttons; RLS policy on update enforces).
